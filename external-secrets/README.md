# External Secrets Operator Configuration

This directory contains the configuration for External Secrets Operator (ESO), which synchronizes secrets from HashiCorp Vault and other sources to Kubernetes Secrets across namespaces.

## Overview

ESO provides:

- **SecretStore / ClusterSecretStore**: Connection configuration to backends (Vault, Kubernetes)
- **ExternalSecret / ClusterExternalSecret**: Declarative mapping from a backend path to a Kubernetes Secret
- **Automated synchronization**: Keeps Kubernetes Secrets in sync with the backend

## Architecture

The cluster uses three real patterns:

### 1. Per-namespace Vault mount (primary pattern)

Each namespace has its own Vault KV v2 mount, its own Kubernetes auth role with a least-privilege policy, and its own in-namespace `ServiceAccount` + `SecretStore` + `ExternalSecret`.

```
Vault (<namespace>/<key>)
    â†“
SecretStore "vault" (in <namespace>) â†’ role "eso-<namespace>" â†’ ServiceAccount "eso"
    â†“

    â†“

```

Examples in the repo: `atc/`, `freqtrade/`, `github-app-shared/`, `github-pat/`, `github-readme-stats/`, `lumos-bot/`. See `../vault/README.md` for the role/policy table.

### 2. Kubernetes provider (postgres-operator DB credentials)

Postgres user secrets generated by the Zalando postgres-operator live in the `postgres-operator-deployment` namespace. ESO uses its Kubernetes provider to copy them into the consuming app namespace.

```
postgres-operator â†’ Secret (postgres-operator-deployment/<user>.<cluster>.credentials.postgresql.acid.zalan.do)
    â†“ (Role + RoleBinding allowing the app's "eso-db" SA to read that one Secret)

    â†“
ExternalSecret (in app namespace)
    â†“
K8s Secret (in app namespace) consumed by the app
```

Used today by `langfuse`, `openwebui`, `mlflow`, and `keycloak`. The `Role` + `RoleBinding` per consumer namespace are defined in `rbac-db-reader.yaml`.

### 3. Cluster-scoped Vault distribution (one secret â†’ many namespaces)

When a single Vault credential needs to be fanned out (read-only) into multiple consumer namespaces, a cluster-scoped store reads it once and a `ClusterExternalSecret` projects the resulting K8s Secret into every namespace matching a label selector. This avoids duplicating SecretStore/SA/role/policy per consumer. App-scoped secrets should still use pattern 1.

```
Vault (github-app-shared/<key>)
    â†“
ClusterSecretStore "vault-github-app" â†’ role "eso-github-app" â†’ ServiceAccount "external-secrets/external-secrets"
    â†“
ClusterExternalSecret (namespaceSelector: secrets-sync/enabled=true)
    â†“
K8s Secret (in every selected namespace)
```

Unlike pattern 1, the Vault role binds to the ESO operator's own SA (`external-secrets/external-secrets`) rather than a per-namespace `eso` SA, because there is no consumer-side SA â€” the operator itself authenticates and writes the Secret into each selected namespace.

Used today: the `github-app-shared` mount holds shared GitHub OAuth credentials that are distributed as the `grafana-github-oauth-secret` Secret into the `grafana` and `monitoring` namespaces (both labeled `secrets-sync/enabled=true`).

#### Onboarding a new consumer namespace

```bash
kubectl label ns <namespace> secrets-sync/enabled=true
```

The operator picks up the new namespace on the next reconcile and creates `grafana-github-oauth-secret` in it.

## Configuration Files

### values.yaml

- ESO deployment configuration
- Resource limits and requests
- Webhook and cert controller settings

### github-app-clustersecretstore.yaml

`ClusterSecretStore` named `vault-github-app`. Reads the `github-app-shared` Vault KV v2 mount and authenticates as the ESO operator SA (`external-secrets/external-secrets`) using the cluster-scoped Vault role `eso-github-app`. Implements pattern 3 above.

### github-app-clusterexternalsecret.yaml

`ClusterExternalSecret` named `grafana-github-oauth`. Selects every namespace labeled `secrets-sync/enabled=true` and produces a `grafana-github-oauth-secret` (Opaque) in each â€” its keys (`dex.github.clientID`, `dex.github.clientSecret`) are sourced from the `k8s-at-shion1305-com` entry under the `github-app-shared` mount.

### rbac-db-reader.yaml

Per-namespace `Role` + `RoleBinding` granting each app's `eso-db` ServiceAccount read access to its own postgres-operator-generated Secret.

### kustomization.yaml

Bundles the above resources for ArgoCD.

## Adding Secrets to a New Application (per-namespace Vault pattern)

### Step 1: Create a per-service Vault mount and write the secret

```bash
# Enable a dedicated KV v2 mount for the service
vault secrets enable -path=<svc> kv-v2

# Write the secret
vault kv put <svc>/credentials API_KEY="..." API_SECRET="..."
```

### Step 2: Create a Vault policy and a Kubernetes auth role bound to the namespace's ServiceAccount

```bash
vault policy write eso-<svc> - <<EOF
path "<svc>/data/*" {
  capabilities = ["read"]
}
path "<svc>/metadata/*" {
  capabilities = ["read", "list"]
}
EOF

vault write auth/kubernetes/role/eso-<svc> \
  bound_service_account_names=eso \
  bound_service_account_namespaces=<svc> \
  policies=eso-<svc> \
  ttl=1h
```

Policies and roles are kept in `vault/scripts/setup-eso-policies.sh` â€” add the new entries there so they can be replayed.

### Step 3: Create the namespace `ServiceAccount` + `SecretStore` and an `ExternalSecret`

```yaml
# <svc>/vault-secret-store.yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: eso
  namespace: <svc>
---
apiVersion: external-secrets.io/v1
kind: SecretStore
metadata:
  name: vault
  namespace: <svc>
spec:
  provider:
    vault:
      server: "http://vault.vault.svc.cluster.local:8200"
      path: "<svc>"
      version: "v2"
      auth:
        kubernetes:
          mountPath: "kubernetes"
          role: "eso-<svc>"
          serviceAccountRef:
            name: "eso"
---
apiVersion: external-secrets.io/v1
kind: ExternalSecret
metadata:
  name: <svc>-credentials
  namespace: <svc>
spec:
  refreshInterval: 5m
  secretStoreRef:
    name: vault
    kind: SecretStore
  target:
    name: <svc>-credentials
    creationPolicy: Owner
    template:
      type: Opaque
  dataFrom:
    - extract:
        key: credentials
```

### Step 4: ESO syncs the Vault data to a Kubernetes Secret in the namespace

ESO authenticates as the `eso` ServiceAccount, exchanges its token for a Vault token via the `eso-<svc>` role, reads the path, and writes the resulting Kubernetes Secret. The application consumes it via `envFrom: secretRef:` or `secretKeyRef`. Existing examples: `atc/external-secret.yaml`, `freqtrade/external-secret.yaml`, `github-readme-stats/external-secret.yaml`.

## Troubleshooting

### Check ESO Status

```bash
# Check ESO pods
kubectl get pods -n external-secrets

# Check (Cluster)SecretStore status
kubectl describe clustersecretstore vault-github-app
kubectl describe secretstore vault -n <namespace>

# Check (Cluster)ExternalSecret status
kubectl describe externalsecret <name> -n <namespace>
kubectl get clusterexternalsecret grafana-github-oauth -o yaml

# View ESO logs
kubectl logs -n external-secrets -l app.kubernetes.io/name=external-secrets
```

### Verify Vault Connection

```bash
# Test Vault connectivity from ESO pod
kubectl exec -n external-secrets <eso-pod> -- \
  curl -k https://vault.k.shion1305.com/v1/sys/health

# Check ESO can authenticate to Vault (look for auth success in logs)
kubectl logs -n external-secrets -l app.kubernetes.io/name=external-secrets | grep -i auth
```

### Force Secret Refresh

```bash
# Annotate the (Cluster)ExternalSecret to trigger an immediate refresh
kubectl annotate externalsecret <name> -n <namespace> \
  force-sync="$(date +%s)" --overwrite
```

## Security Considerations

- **Namespace isolation**: Each namespace has its own `SecretStore` + `ServiceAccount` + Vault role
- **Least privilege**: Per-namespace Vault policies restrict reads to only that namespace's paths
- **Service account binding**: Each Vault role is bound to a specific ServiceAccount in a specific namespace
- **Token lifecycle**: Automatic token renewal and rotation

## Auto-restart on Secret Changes

To automatically restart pods when secrets change, use Stakater Reloader:

```yaml
# Add to your Deployment
metadata:
  annotations:
    reloader.stakater.com/auto: "true"
    # Or for a specific secret:
    # secret.reloader.stakater.com/reload: "<secret-name>"
```
